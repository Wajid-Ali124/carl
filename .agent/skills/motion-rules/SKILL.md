# Motion (Framer Motion) + Next.js Rules

- **⚠️ PRIORITY — Wrap entire app in `LazyMotion` in `layout.tsx` and replace every `motion.*` with `m.*`.** Default `motion` component = ~34KB. `LazyMotion + domAnimation + m` = ~4.6KB initial. Highest-impact change available. Add `strict` prop in dev to catch accidental `motion.*` slipping through. For maximum score, async-load features so animation JS defers until after hydration: create `lib/motionFeatures.ts` exporting `domAnimation`, then pass `features={() => import("../lib/motionFeatures").then(r => r.default)}` to `LazyMotion`. Only use `domMax` on pages that actually need drag or layout animations — not globally.
- **Every file using motion components needs `"use client"` at the top.** Motion uses browser APIs unavailable during SSR. Missing this in App Router causes silent runtime failure or hydration crash. Layout animations (`layout`, `layoutId`) also cause `data-projection-id` server/client mismatch — fix with `suppressHydrationWarning` on the element or a `useEffect`/`useState(false)` client gate.
- **Only animate `x, y, scale, rotate, skew, opacity`.** These are compositor-only, GPU-handled, zero layout cost. Never animate `width`, `height`, `top`, `left`, `margin`, `padding` — every frame triggers full layout recalc. Exception: `height: "auto"` for accordions is safe — Motion handles it via FLIP internally. For scroll-linked `useMotionValue`/`useTransform` animations, add `style={{ willChange: "transform" }}` — but don't apply `willChange` broadly, each creates a new GPU layer.
- **⚠️ Always set `viewport={{ once: true }}` on every `whileInView`.** Without it, Motion mounts/unmounts IntersectionObserver callbacks on every scroll entry/exit across all animated elements — tanks INP on scroll-heavy pages. Also: `whileInView` + `transition: { repeat: Infinity }` is a known bug and simply doesn't work. For infinitely looping in-viewport animations, use `onViewportEnter`/`onViewportLeave` callbacks with `useAnimate` instead.
- **⚠️ Remove all Tailwind `transition-*` classes (`transition-all`, `transition-transform`, `duration-300`) from any Motion-animated element.** Tailwind CSS transitions conflict with Motion's inline style animations causing stuttery or broken movement. Official rule: Tailwind handles style, Motion handles motion — never both on the same property.
- **Use variants for site-wide stagger** — define once, reuse everywhere. Parent `staggerChildren` propagates to all children automatically, no per-child `delay` props needed. Wrap `AnimatePresence` around any conditionally rendered element and always pass a unique `key` — without it, exit animations never play. Use `mode="wait"` for page transitions, `mode="popLayout"` for list reordering.
- **For 50+ simultaneously animated items, use `react-window` or `react-virtuoso`.** Motion only animates rendered rows — without virtualization, animating hundreds of items causes severe jank. For scroll-linked parallax, `useScroll` + `useTransform` update motion values directly without triggering React re-renders — zero state overhead per scroll event.
- **Add `<MotionConfig reducedMotion="user">` once at the app root.** Automatically disables transform and layout animations for users with system Reduce Motion enabled — no per-component `useReducedMotion` hooks needed. Never set initial state via Tailwind classes (e.g. `opacity-0`) — if Motion's trigger misses, element stays invisible permanently. Always use Motion's `initial` prop instead.
- **Package was renamed: `framer-motion` → `motion`, import from `"motion/react"`.** Old package still works — APIs are identical. Do not mix both in the same project. Motion is rarely the actual bottleneck for sub-90 scores — fix these first: `next/image` with correct dimensions (unoptimized images = #1 LCP killer), `next/font` (removes render-blocking Google Fonts), third-party scripts with `strategy="lazyOnload"`, and keeping heavily animated sections in separate client components so they don't bloat the initial JS chunk.
